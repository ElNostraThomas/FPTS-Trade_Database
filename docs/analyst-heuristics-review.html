#1
FAAB → MVS ratio (× 10)
Hand-tuned
Source location
trade-calculator.html:2346
Current math
side.faab * 10
Reasoning
The 10:1 ratio approximates the dynasty community heuristic that $1 of FAAB ≈ 10 mid-range trade-value units (e.g. $100 FAAB ≈ 1000, which is around a late-3rd / early-4th rookie pick). In auction leagues or shallow leagues this ratio breaks down.
Should this scale with league depth (starter count) or FAAB budget? In mixed-auction leagues this ratio may not hold.

#2
Age-curve constants per position
Hand-tuned
Source location
assets/js/player-panel.js:908
Current values
RB:  peak 24, peakVal ~8500, decay 0.78/yr  (~22% drop per year past peak)
WR:  peak 26, peakVal ~9000
TE:  peak 27, peakVal ~8500
QB:  peak ~26, peakVal ~9000, decay 0.88/yr (~12% drop per year past peak)
Reasoning
Peak ages roughly match community-cited positional aging research (RB peak earliest at 24, TE latest at 27). Decay rates are hand-tuned to match observed "the cliff" patterns. PeakVals are intentionally close (~8500–9000) so a "max-tier WR" and "max-tier QB" feel comparable.
Regression against actual dynasty value-vs-age data could refine these constants. Not currently regressed against any real dataset.
Recommended replacement
cross-ref: age-curves-per-pos
#3
ADP-implied value decay (0.93/pick)

maxVal × 0.93^(adp - 1)
Sample outputs

Slot 21   →  2342

Reasoning

#4
Buy-low / sell-high threshold (±500)
Hand-tuned
Source location
assets/js/player-panel.js:984
Current math
gap > 500 → "buy"; gap < -500 → "sell"
Reasoning
±500 represents a "meaningful gap" — roughly 5–7% of a mid-range dynasty asset value. Below that the FC/ADP delta is plausibly noise.
Percentage-based threshold (e.g. ±10%) would scale better. At very high values (10000) the current 500 threshold is only 5%; at low values (2000) it's 25%. The asymmetry probably under-fires for stars and over-fires for depth pieces.
Recommended replacement
cross-ref: buy-sell-signal
#5
Composite archetype weights (0.6 / 0.2 / 0.2)
Hand-tuned
Source location
my-leagues.html:2942
Current math
composite = 0.6*valNorm + 0.2*pickNorm + 0.2*projNorm
Reasoning
Player value gets 3× weight because skill-position rosters are the dominant indicator of dynasty league strength (vs picks which are future-tense and projections which are short-tense). 20/20 split between picks and projections reflects them being roughly equally informative for "where this team is right now" (picks lean future, projections lean current).
Sensitivity analysis — does shifting to 0.5/0.3/0.2 or 0.7/0.15/0.15 change archetype assignments meaningfully?
Recommended replacement
cross-ref: team-archetype
#6
Archetype value/age cutoffs (±10% value, ±0.6yr age)
Hand-tuned
Source location
my-leagues.html:2944–47
Current math
valueHigh = composite > 1.10
valueLow  = composite < 0.90
ageBand   = ±0.6yr from league median
Reasoning
±10% composite value creates 3 buckets (low / mid / high) that roughly correspond to bottom-3 / middle-6 / top-3 in a 12-team league. ±0.6yr age band is intentionally narrow to catch real youth/age skews (the median league age band is typically ~3 years total, so ±0.6 = ~40% of the spread).
Border teams (composite 1.09 → tweener vs 1.11 → contender) flip on small changes. Should there be a "soft border" treatment or a wider band?
Recommended replacement

#7
Trade-suggester weights (0.55 / 0.35 / 0.10)
Hand-tuned
Source location
my-leagues.html:3796
Current math
fitScore = 0.55*valueFit + 0.35*archFit + 0.10*fewer
Reasoning
Value-fit dominates because a trade that's far from par is broken regardless of archetype alignment. Archetype-fit is secondary (35%) because a fair trade that doesn't match the target's preferences is unlikely to get accepted. The 10% "fewer assets" weight is a mild penalty for multi-asset packages — keeps the simpler offer ranking higher when scores are otherwise close.
Are the weights right? Suggester-acceptance data (which suggestions actually got traded) would refine these.
Recommended replacement
cross-ref: trade-suggestion-fit

#9
Heatmap text-contrast threshold at 35%
Hand-tuned
Source location
assets/js/heatmap.js
Current math
textColor = v >= 35 ? "#111" : "rgba(255,255,255,.5)"
Reasoning
At ~35% alpha on the brand orange (rgb 237,129,12), the bg becomes "dark enough" that #111 text is more readable than semi-white. Below 35% the bg is faint enough that semi-white reads better on the parent surface. Picked by eye.
Verify against WCAG AA contrast ratio. The 35% threshold could be derived from the actual bg luminance vs both candidate text colors.
Recommended replacement

#10
MVS overlay zeros out FP players not in MVS
Intentional
Source location
assets/js/data-bootstrap.js:191–194
Current math
Object.values(FP).forEach(rec => {
  if (overlaid.has(rec)) return;
  rec.value = 0; rec.valueSf = 0; rec.value1qb = 0;
});
Reasoning
Intentional. Players in data/values.json without a matching MVS record are most likely inactive / cut / suspended — keeping the stale FP value would mislead users (e.g. showing a 5000-value chip on a player who hasn't played in 6 months). Zeroing makes them obviously "no current market" in any value display.
No action required. Documented here so anyone reading the code understands why stale FP-only players display as $0 across the calc instead of falling back to their last known FP value.
no cross-refs
#11
3-way trade balance uses different denominator
Code bug
Source location
trade-calculator.html:2497
Current math
2-way: pctOff = abs / max(t0, t1)        // leader-relative
3-way: pctOff = gap / sorted[0].total    // top-relative
Reasoning
Likely an oversight from when 3-way support was added — code re-used the leader-relative pattern instead of generalizing the max-relative pattern. 3-way trades are rare in calc usage data so the bug rarely surfaces.
Code-correctness fix, not a math-validity question. Generalize the 2-way pattern so both branches use the same denominator max(t0..tN). Likely a 1-line change.
Fix proposal
cross-ref: trade-balance
#12
MVS valueTep aliased to valueSf (no real TEP)
Intentional
Source location
assets/js/data-bootstrap.js:203
Current math
valueTep: rec.valueSf // alias — no real TEP in upstream CSV
Reasoning
The upstream MVS CSV (player_market_mvs.csv) only ships value_sf and value_1qb columns for picks — no TEP variant. Aliasing valueTep to valueSf means: in SF+TEP mode, an MVS pick shows the SF value (the closer of the two available numbers). The picks.json (FP API, 2025-only) DOES have real TEP data; for 2026–2028 picks coming via MVS, we don't.
No action unless the upstream CSV gains a TEP column. If/when it does, drop the alias and read rec.valueTep directly. Tracked so the alias doesn't get treated as a real value source.
cross-ref: pick-numeric-value
#13
leagueAvg is a median, not a mean
Naming bug
Source location
my-leagues.html:5093–5174
Reasoning
Earlier code used means; later refactored to medians (more robust to outliers in 12-team leagues where 1 standout team can skew the mean). The variable name didn't get updated. Pure naming bug, no math impact.
Rename leagueAvg → leagueMedian everywhere in the file so anyone reading the code doesn't assume mean. Safe global find-replace.
Fix proposal
cross-ref: league-medians
#14
valColor has duplicate green branches
Cosmetic artifact
Source location
my-leagues.html:6292
Current math
v > 8000  → green     // originally bright-green
v > 5000  → green     // originally light-green mid-shade
v > 2000  → green-bg
otherwise → muted
Reasoning
Originally v > 8000 → bright-green and v > 5000 → light-green mid-shade. The brand-color audit (commit fca6d0e) collapsed the mid-shade to brand green for palette consistency. The duplicate branches were left in case the design wants to differentiate again later. Effectively one threshold at v > 2000 (everything above is green).
Either (a) collapse to a single v > 2000 branch and delete the dead conditional, or (b) re-introduce a distinct mid-shade green for the 5000–8000 band if the design wants more granularity.
Decision
